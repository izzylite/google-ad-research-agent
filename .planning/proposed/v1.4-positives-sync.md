# v1.4 — Positives Sync (proposed)

Status: parked. Waiting on v1.3 (Phase 12 Tavily removal) to ship.
When ready: feed this doc into `/gsd:new-milestone v1.4 — Positives Sync`.

---

## Problem

Skill re-runs on same client produce ranked keyword lists that overlap with
keywords the account already runs. Operator manually scrubs duplicates before
pasting `positives.csv` into Google Ads Editor. Tedious, error-prone, and
asymmetric with negatives (Phase 8 already dedups negatives via `negatives-sync.json`).

## Goal

Mirror negatives-sync for positives. Diff ranked keywords against the client's
currently-active Google Ads keywords; surface only net-new in `positives.csv`.

## Scope

### Phase 14 — Positives Sync (single phase, ~half-day)

> Phase 13 is already claimed for Landing-Page Extract Vendor Swap
> (defer-until-friction backlog under v1.3). Positives Sync is the next
> available number under v1.4.

Three small additions, no new external APIs:

1. **`perf_fetch.py` — new GAQL query** (~50 LOC)
   - Query `keyword_view` for active + paused keywords in last 30d
   - Pulls: kw text, match_type, status, impressions, clicks, conversions, cost
   - Skip Performance Max campaigns (no kw-level data exists for PMax)
   - Writes `raw/google-ads-keywords.json`

2. **`perf_synth.py` — cross-ref function** (~80 LOC)
   - `cross_ref_positives(ranked, existing_kws)` returns 4 buckets:
     - `already_active` — exact + lemma-hash match, status ENABLED
     - `paused_in_account` — match, status PAUSED (reactivate candidate)
     - `covered_by_broad` — ranked exact within broad-match scope of an active kw
     - `new_to_add` — net-new
   - Writes `positives-sync.json` mirroring negatives-sync structure
   - Stats line: our_total / already_active / paused / covered_by_broad / new_to_add

3. **`render_report.py` + `export_csv.py`** (~75 LOC)
   - `render_positives_sync_section()` — md + html, mirrors negatives-sync section
   - Stats line surfaces all 4 buckets
   - Enumerate `new_to_add` full kw list; `paused` + `covered_by_broad` as count + collapsible
   - `already_active` = count only (mirror negatives-sync behavior)
   - `positives.csv` default filter: `new_to_add` only
   - `--include-existing` CLI flag on export for full list

### Optional LLM reasoning layer

After script-based dedup, LLM re-reads `new_to_add` + `already_active` and re-tags
borderline cases (semantic dupes, match-type drift). Catches ~20% script misses.
SKILL.md addition ~30 LOC.

## Requirements (proposed)

| ID | Description |
|---|---|
| POS-01 | `perf_fetch.py` pulls `keyword_view` last 30d; writes `raw/google-ads-keywords.json` |
| POS-02 | `perf_synth.py` produces 4-bucket `positives-sync.json` |
| POS-03 | Report `## Positives Sync` section renders 4 buckets with mirrored UX |
| POS-04 | `positives.csv` filters to `new_to_add` by default; `--include-existing` overrides |
| POS-05 | Phase 13 graceful-skips if `raw/google-ads-keywords.json` absent (no OAuth) |
| POS-06 | LLM reasoning step in SKILL.md re-tags borderline semantic dupes (optional) |
| POS-07 | Test coverage: cross_ref_positives unit tests + golden positives-sync.json fixture |

## Anti-goals

- No new external APIs (use existing GAds OAuth from Phase 8)
- No Ahrefs paid-kw cross-ref (separate phase if Phase 8 OAuth unavailable)
- Don't touch ranking logic — sync is post-rank filter only
- Don't auto-pause kw already running — surface, let operator decide

## Risks

- **Match-type semantics nuance**: Google's broad-match expansion is fuzzy; our
  `covered_by_broad` heuristic will have false positives. Mitigation: LLM
  re-tag step catches obvious misses; surface bucket as "review" not "skip".
- **Smart Bidding auto-added keywords**: PMax + Search Themes add kw the operator
  didn't explicitly choose. `keyword_view` shows them; we'll treat as active.
  Possible operator confusion. Surface origin in HTML where available.
- **Account access scope**: requires OAuth on target account. Already a Phase 8
  prereq, so no new auth burden.

## Out of scope

- Bid-conflict detection (existing kw at higher CPC than our suggested)
- Match-type auto-tightening recommendations
- Negative-keyword conflict detection on net-new (already handled at Editor import time)
- Ahrefs paid-kw fallback (separate phase)
