# v1.5 — Account-Aware Narrowing (proposed)

Status: parked. Two related issues surfaced during Lake Worth dogfood
(2026-05-15), both stemming from the same architectural gap: **skill output
narrows to the OAuth scope (whole account) instead of the operator's actual
target (one campaign + the AGs inside it)**.

Phase 11 added `geo_focus` to brief.md for **geographic** narrowing. v1.5
extends the same pattern to:

1. **Campaign focus** — `campaign_focus` brief field filters all Phase 8 GAQL
   queries to a single campaign
2. **AG mapping algorithm enrichment** — Jaccard input grows from AG name
   only to AG name + kw_criteria + search-term bag (raises coverage from 0%
   to 50%+ on short-name-AG accounts)

When ready: `/gsd:new-milestone v1.5 — Account-Aware Narrowing`.

---

## Issue 1 — Campaign Focus

### Problem

Phase 8 `perf_fetch.py` GAQL queries have no campaign filter. One Google Ads
account (`GOOGLE_ADS_CUSTOMER_ID`) contains many campaigns. Operator's brief
typically targets ONE campaign (e.g. `Search | Lake Worth Accident Exams |
Manual CPC`), but skill pulls all 30+ campaigns + 35+ ad groups → cross-refs
get contaminated with irrelevant data:

| Section | Contamination |
|---|---|
| Existing Ad Groups list | Shows AGs from Palm Springs Car Accidents, Hybrid Car Accident, FL PIP Exams etc. when operator only cares about Lake Worth Accident Exams |
| Positives Sync `already_active` | Includes kw running in unrelated campaigns — operator might wrongly skip them as duplicates |
| Negatives Sync `already_in_account` | Same issue — negs from other campaigns inflate count |
| Ad Group Mapping | 35 AGs as match candidates instead of 3 → dilutes Jaccard signal |

### Goal

Single source of truth for campaign focus: brief.md `campaign_focus:` line.
All Phase 8 GAQL queries inherit `AND campaign.name = '<focus>'` filter when
set. Downstream Positives Sync + Negatives Sync + Ad Group Mapping respect
the narrowed dataset automatically.

### Scope

#### Phase 15 — Campaign Focus

1. **`brief.md` schema**: new optional field `campaign_focus:` (single value
   or list — TBD per gsd:discuss-phase). Parsed in `_parse_brief_fields`
   alongside existing `geo_focus`.

2. **`perf_fetch.py`**: 4 GAQL queries (`keyword_view`, `search_term_view`,
   `ad_group`, `campaign_criterion`) accept `--campaign-filter` CLI arg
   (auto-populated from brief.md). Filter clause:
   `AND campaign.name = '${focus}'` (single) or
   `AND campaign.name IN ('${a}','${b}')` (list).

3. **Graceful degrade**: when `campaign_focus` absent → current account-wide
   behavior. Phase 14 (Positives Sync) and Phase 11 (AG Mapping) inherit
   filtered raw data without code changes since they consume `raw/*.json`.

4. **Render**: report header "Campaign Focus" callout sits beside Geographic
   Focus. Validates focus name appears in `raw/google-ads-perf.json` campaigns
   list (catches typos).

### Requirements

| ID | Description |
|---|---|
| CAMP-01 | `_parse_brief_fields` extracts optional `campaign_focus` (single or list) |
| CAMP-02 | `perf_fetch.py --campaign-filter '<name>'` filters all 4 GAQL queries |
| CAMP-03 | SKILL.md Phase 8 step auto-passes `campaign_focus` from brief.md to perf_fetch |
| CAMP-04 | Graceful degrade: no `campaign_focus` → account-wide pull (current behavior) |
| CAMP-05 | Report header renders Campaign Focus callout when set; validates against raw perf.json |
| CAMP-06 | Test coverage: golden brief w/ campaign_focus + respx assertion on GAQL filter clause |

### Anti-goals

- No campaign **inference** from brief content (operator names campaign explicitly)
- No Account-wide vs campaign-scoped TOGGLE — campaign_focus presence is the toggle
- No PMax filtering exception — PMax campaigns surface in account-wide queries; operator excludes via `campaign_focus`

### Risks

- **Campaign name typos**: GAQL returns empty silently. Mitigation: CAMP-05 validates name appears in perf.json campaigns list before applying filter; warn operator on typo.
- **Multi-campaign briefs**: rare but real (operator running Lake Worth across 2 campaigns). List support handles this; single-value default keeps common case clean.
- **MCC accounts**: campaign_focus still applies per `GOOGLE_ADS_CUSTOMER_ID`; MCC-level rollup out of scope.

## Issue 2 — Ad Group Mapping Algorithm Enrichment

### Problem

`ad_group_match.py` Jaccard token overlap currently uses **ad group NAME**
as the only evidence for matching. Real client ad group names are short
labels (`Accident Exams – Lake Worth`, `Palm Springs`, `RSA - Lake Worth`);
ranked keywords are long phrases (`auto accident injury treatment wellington`).

Token overlap stays below 0.4 floor for nearly every pair → all kw land in
`low` confidence tier → 0% high+medium coverage → operator can't auto-route.

Lake Worth dogfood run (2026-05-15): **0% coverage on 83 ranked kw vs 35
existing ad groups**. STATE.md notes this is mathematically correct, but
operator-painful.

### What post-v1.4 cosmetic fix already did

Commit `4674b00`: mapping section always lists all existing ad groups
regardless of match quality, so operator at least sees their account
structure. Algorithm still misses 100% of matches in short-name-AG accounts.

### Goal

Lift coverage on real client accounts from 0% to 50%+ by feeding the matcher
richer evidence per ad group:

- Active **keyword criteria** in the ad group (from Phase 14 `keyword_view`)
- Top **search terms** that triggered ads in the ad group (from existing
  Phase 8 `search_term_view`)

Both already in `raw/` — no new API calls needed.

### Scope

#### Phase 16 — Ad Group Token-Bag Enrichment

1. **`ad_group_match.py`**: new `_build_ag_token_bag(ag_name, kw_criteria,
   search_terms)` — token bag = AG name tokens ∪ kw_criteria tokens ∪
   top-10 search-term tokens. Replaces `_tokens(ag.name)` as Jaccard input.

2. **Threshold recalibration**: token bag grows from ~4 tokens (AG name only)
   to ~30-100 tokens. Existing thresholds (0.7 high / 0.4 medium) may need
   re-tuning — likely tighter (e.g. 0.5 high / 0.25 medium) because larger
   bags inflate denominator. Calibrate on Lake Worth + 1-2 other real accounts.

3. **Confidence rationale extension**: `match.reason` field captures which
   bag(s) contributed (`"jaccard=0.32 on kw-criterion bag, AG name overlap 0"`)
   so operator can audit.

4. **Backward compat**: when `raw/google-ads-keywords.json` (Phase 14 output)
   absent, fall back to current name-only Jaccard. Phase 11 behavior preserved
   for pre-Phase-14 runs.

### Requirements

| ID | Description |
|---|---|
| ADGM-07 | `_build_ag_token_bag` produces token set from AG name + kw_criteria + top-N search-terms |
| ADGM-08 | Jaccard scoring uses enriched token bag; falls back to name-only when Phase 14 raw absent |
| ADGM-09 | Match `reason` field surfaces which evidence source contributed |
| ADGM-10 | Threshold recalibration documented in `references/phase11-account-structure-mapping.md` |
| ADGM-11 | Test coverage: golden mapping fixture from Lake Worth real-account data; assert ≥50% coverage |

### Anti-goals

- No new external APIs (reuse Phase 8 + Phase 14 raw data)
- No LLM-based matching (deterministic Jaccard math stays auditable)
- No ad group performance weighting (separate concern — ROAS prioritization)

### Risks

- **Threshold drift**: bigger bags → larger denominators → score compression. Mitigation: empirical calibration on 2+ real accounts before locking.
- **Search-term noise**: lossy / off-topic search terms could pull matches wrong direction. Mitigation: cap to top-10 by clicks; filter zero-impression.
- **Compute cost**: 83 kw × 35 AGs × (name + kw-criteria + search-terms tokenization per AG) = ~2900 Jaccard calls. Still sub-second on practical account sizes.

## Cross-Issue Notes

- **Phase ordering**: Phase 15 (Campaign Focus) lands FIRST. Phase 16 (AG token-bag enrichment) consumes the narrowed dataset Phase 15 produces. Reversing the order means token-bag enrichment runs against the full account dataset → noisier calibration.
- **Combined effect**: with both shipped, Lake Worth dogfood should see (a) only the 3 ad groups inside `Search | Lake Worth Accident Exams | Manual CPC`, AND (b) 50%+ of ranked kw map to those 3 ad groups with high/medium confidence. Operator pastes new kw directly into existing AGs without re-thinking structure.

## Out of scope (both issues)

- Performance-aware match weighting (high-ROAS AGs win ties) — defer until calibration data exists
- LLM semantic similarity (cost + non-determinism)
- PMax campaign handling (PMax has no kw-criterion)
- Auto-create new ad groups in the operator's account — skill stays read-only / suggest-only

## Evidence from Lake Worth run

- `positives-sync.json` (Phase 14): 11 already_active + 8 covered_by_broad = 19 matches at kw-text level — proves the data is present to support Issue 2's token-bag approach.
- `report.html` post-fix screenshot: 34 ad groups listed, span 30+ campaigns including 5+ irrelevant ones (Palm Springs, FL PIP, Hybrid Car Accident...) — proves Issue 1's contamination is real and visible to operator.
